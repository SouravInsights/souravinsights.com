Building a photo uploader seems like a solved problem. You drop an `<input type="file" multiple />` into your React component, wire up a `fetch` request, and call it a day. If you're building a simple avatar uploader, that's exactly what you should do.

But reality fractures when you try to scale that naive approach. When building the **TripMediaLibrary** for BeenThere, our constraints were entirely different. Users needed to upload batches of high-resolution trip photos directly from their phones. We have a hard cap of 1GB space allocation for free users and a maximum of 20 photos per trip. These aren't heavily compressed web images; these are pristine, 8MB Apple HEIC and raw camera files.

When you try to run twenty 8MB images through a standard JavaScript `Promise.all()` loop on a mobile device, things break. The UI stutters. The browser memory balloons until the tab silently crashes. Progress bars jump erratically, and if the network blips, the entire experience falls apart.

To make the upload process feel intentional, reliable, and fluid—complete with a skeuomorphic UI that visually "develops" polaroids as they upload—we had to completely rethink how we handle data in the browser. 

Before we dive into the code, here is the complete 4-phase architecture we built to solve this. Click through the phases to build a mental map of how the data flows before we unpack the theory below.

## Why the browser begs for mercy

When building a modern photo uploader, you don't just blindly send files to a server. You want to do local processing first—extracting EXIF data (for GPS coordinates and capture dates) and generating client-side thumbnails to give the user immediate visual feedback.

If you attempt to run this kind of heavy processing directly in your React components, your application will freeze.

To understand why, we have to look at how JavaScript executes. The JavaScript engine running your UI is strictly **single-threaded**. It has one call stack and one memory heap. It processes tasks via the Event Loop. When you use `async/await` to process multiple images concurrently, you are not actually doing the work simultaneously. The Event Loop is simply interleaving chunks of work, switching back and forth between them so fast that it *looks* like they are happening at the same time.

But parsing heavy binary data or running a JavaScript-based image decoder is CPU-intensive, synchronous work. If you hand the main thread an 8MB file and ask it to find the EXIF headers, it has to block the call stack to do the math. While the call stack is blocked, the browser cannot paint new frames to the screen. It cannot respond to the user clicking a "Cancel" button. It cannot even animate a simple CSS spinner. The entire application locks up.

The benchmark below runs the real pipeline—actual EXIF parsing with `exifr` and real HEIC decoding with `heic2any`. Drop some photos in and run both modes. The most telling number isn't whether the spinner freezes; it's the wall-clock difference in the total time. On the machine this article was written on, processing 12 HEIC images took **20.3 seconds on the main thread** versus **7.8 seconds with a worker pool**—a 2.6× speedup, while the UI remained completely unblocked throughout.

<HeicBenchmark />

Your numbers will vary depending on your device, but the shape of the result won't. The main thread processes images serially, one blocking operation after another. The worker pool saturates your CPU cores and processes images in parallel, finishing in roughly `total_work / core_count` time. On a high-end desktop the difference might feel modest. Run it on a mid-range Android phone and the main thread version will either crawl or crash the tab entirely—the worker version will finish cleanly in the background while the UI stays interactive the whole time.

This gap is the entire reason Web Workers exist. The main thread is a shared resource between your JavaScript, your React reconciler, your CSS animations, and the browser's own painting pipeline. Handing it a 200MB decode job is like asking a restaurant's maître d' to also personally cook every dish. The fix isn't to cook faster—it's to route the work to a kitchen that runs independently.

## Escaping the main thread

To fix this, we didn't just need concurrency (interleaving tasks on one thread); we needed true **parallelism** (executing tasks simultaneously on different CPU cores). 

The only way to achieve true parallelism in the browser is by spawning an isolated OS-level thread via a **Web Worker**.

Instead of doing the heavy lifting in our React components, we built a dedicated worker pool. When the user drops their photos into the UI, the main thread immediately intercepts the `File` objects and uses `postMessage` to hand them off to our background workers. 

Our pipeline requires two distinct trips to the worker pool.

### The first trip: Extraction and normalization
Our first worker (`extract.worker.ts`) is responsible for parsing the EXIF data and preparing the file for the browser. 

Because we expect a lot of Apple users, we have to deal with the `.HEIC` image format. Most browsers cannot natively render HEIC files in an `<img>` tag. If we try to show a preview of an HEIC file, it will just show a broken image icon. 

To solve this, our worker uses a library called `heic2any` to safely decode the HEIC binary and convert it into a standard JPEG. By running this in an isolated thread, the heavy decoding process utilizes a separate CPU core, leaving the main thread buttery smooth. Once the worker is done, it posts the standard JPEG back to the main thread, which instantly renders a crisp UI preview.

### The second trip: WASM compression
Uploading a 20-photo batch of raw 8MB files would consume 160MB of the user's data plan and heavily tax our backend. We needed to compress the images on the client side *before* they hit the network.

When the photos are ready for upload, the main thread sends them to our second worker (`compress.worker.ts`). This worker spins up an `OffscreenCanvas` to resize the images down to a maximum of 1920x1920.

But standard HTML Canvas compression isn't great. To get the absolute best quality-to-size ratio, we bypass the browser's built-in encoders entirely. Instead, the worker dynamically lazy-loads `@jsquash/jpeg`, an incredibly powerful MozJPEG encoder compiled into **WebAssembly (WASM)**. WebAssembly allows us to run near-native C/C++ compression algorithms directly inside the browser's worker thread.

Simultaneously, the worker generates a `blurhash`—a tiny 32x32 encoded string that represents the blurred colors of the image. This string is so small it can be saved directly in our database, allowing us to show instant, beautiful placeholders when someone visits the trip page later.

## The unseen network traffic jam

At this point, we have highly optimized JPEGs ready to fly to our Cloudflare R2 storage buckets. 

The naive approach is to map over our 20 files and fire off twenty `fetch()` requests at the exact same time. But the network is not a magical, infinite pipe. It is constrained by physical hardware and browser limitations.

If you attempt to open 20 simultaneous TCP connections to the same domain, every modern browser will step in and stop you. Browsers typically enforce a hard limit of around **6 concurrent connections per domain**. If you exceed this, the browser simply throws the excess requests into a hidden internal queue. You lose all control over the upload priority, and your application's other API requests (like fetching the user's profile data) will be blocked behind the massive image uploads.

Worse, attempting to upload 20 massive payloads concurrently requires the browser to hold all of those binary buffers in the device's RAM at the same time. On a desktop, this might be fine. On a mobile browser with strict memory limits, holding that much data in active memory will trigger an Out Of Memory (OOM) exception, and the OS will ruthlessly kill the browser tab.

## Controlling the floodgates

To survive the hostile environment of the browser network stack, we must control the concurrency ourselves. We implemented a **Semaphore**—a computer science synchronization primitive that strictly limits how many tasks can access a resource at once.

We set our magic number to `3`. At any given moment, exactly three photos are allowed to acquire a network slot and begin uploading.

```typescript
const MAX_CONCURRENT_UPLOADS = 3;
let activeUploads = 0;
const queue = [];

export function acquireUploadSlot(): Promise<() => void> {
  // Fast path: free slot available right now.
  if (activeUploads < MAX_CONCURRENT_UPLOADS) {
    activeUploads += 1;
    return Promise.resolve(createRelease());
  }

  // Slow path: queue until an active upload releases a slot.
  return new Promise((resolve) => {
    queue.push(resolve);
  });
}
```

Before a photo is allowed to ask our server for an R2 upload URL, it must `await acquireUploadSlot()`. If three photos are already uploading, the fourth photo's promise remains unresolved. It simply waits in the queue, completely dormant, consuming zero network resources. 

When one of the active uploads finishes, it calls `release()`. This decrements the active count and instantly resolves the promise for the next photo in the queue. 

By strictly chunking the uploads into groups of three, we keep our memory footprint tiny, we respect the browser's connection limits, and we guarantee that our upload pipeline never crashes the user's tab.

## When the network is too fast

With the heavy lifting offloaded to workers and the network safely throttled by a semaphore, the backend architecture was rock solid. But we ran into a bizarre User Experience problem on the frontend.

Our UI was designed to show a vintage polaroid sliding out of a camera and dropping into a stack when an upload finished. 

If a user was on a gigabit fiber connection, the upload would finish in a fraction of a second. Because our UI state was strictly bound to the network state, the polaroid would flash onto the screen for a single frame and instantly disappear. It looked exactly like a rendering glitch.

Human perception is slow. We need time to process visual changes. To fix the "glitch," we had to deliberately **decouple our visual state from our network state**.

In our React orchestration hook, we introduced a visual gate. We track the exact millisecond the polaroid animation starts. Even if the network upload finishes instantly, the code forces an artificial delay to ensure the polaroid has been visible for a minimum of 1.3 seconds before it drops onto the stack. This guarantees the CSS animations always have enough time to play out beautifully, regardless of how fast the user's internet is.

## The silent killer: Render thrashing

As the photos flow through the pipeline, they emit constant progress events: 10%... 12%... 15%... 

If you take those progress numbers and throw them directly into React's `useState()`, you will destroy your application's performance.

When you call `setState()`, React schedules a re-render. It has to reconcile the component tree and figure out what DOM nodes need updating. If you have 3 photos uploading concurrently, each emitting progress events every few milliseconds, React is going to try to re-render your entire upload component hundreds of times a second. 

This is called **Render Thrashing**. The main thread will spend all of its time calculating React diffs, causing your smooth polaroid CSS animations to aggressively stutter and drop frames.

To protect the UI, we aggregated the progress using **Mutable Refs**.

A `useRef` acts like an escape hatch from React's reactive data flow. When you mutate a ref, React does *not* trigger a re-render. 

As the constant stream of progress numbers pours in from the network, we silently update a `Map` stored inside a `useRef`. We then calculate the *overall* batch percentage. We only push that final, aggregated percentage into `useState` to update the global progress bar.

By buffering the aggressive network events into a silent reference and severely throttling the actual state updates, we keep the React render cycle perfectly calm.

## Bringing it all together

Building a production-grade uploader isn't just about calling `fetch`. It's a continuous balancing act between the limitations of the hardware, the restrictions of the browser, and the expectations of the user. 

By pushing CPU-bound work into isolated threads, strictly controlling network concurrency with Semaphores, and fiercely protecting the React render cycle from thrashing, we built a pipeline that feels invisible. And in frontend engineering, an invisible pipeline is exactly what you want.
